## 🌍 世界观

### 动态自适应协作（Oh My OpenCode 风格）

**核心理念**：代理自主决策委派，而非静态依赖图

```
任务启动
    ↓
Agent 自主分析
    ↓
决策：我能独立完成吗？
    ├─ 是 → 自主执行
    └─ 否 → 动态委派给其他 Agent
        ↓
    协作完成
```

**关键特征**：

- ✅ **Ask @oracle 模式**：代理可以主动请求外部专家（GPT、Gemini）协助
- ✅ **实时调整**：根据任务进展动态调整协作策略
- ✅ **零预设图**：不依赖预定义的 blockedBy/blocks 关系

## 📜 最高准则

### 1. 测试铁律

**写功能必须写测试，测试不绿不准 commit**

```
写代码（几行）→ 写对应测试 → 跑测试
  ├─ 红灯 → 分析错误 → 修复 → 再跑（最多5轮）
  ├─ 绿灯 → git commit（原子提交≤50行）
  └─ 5轮修不好 → 停下来重新分析
```

### 2. 原子提交

**单次 commit 改动不超过 50 行，每个 commit 只改一件事**

- ✅ **禁止混合提交**：不要在同一个 commit 中做多个功能
- ✅ **测试必须过**：不测试 = 埋雷
- ✅ **禁止绕过**：不准注释测试、降低断言、跳过用例

### 3. Ralph Wiggum Loop

**循环审核直到满意**

```
完成任务 → 自动审核
    ↓
满意吗？
    ├─ 否 → 修复 → 再审核
    └─ 是 → 标记完成
```

**上限**：最多 5 次自修，超过则上报人工

## 🧠 思维链模板

### 系统化问题分析

```markdown
1. 我面临什么问题？
   - 现象描述
   - 影响范围
   - 紧急程度

2. 问题根源是什么？
   - 技术原因
   - 流程原因
   - 人为原因

3. 有哪些解决方案？
   - 方案 A：[描述]
   - 方案 B：[描述]
   - 方案 C：[描述]

4. 哪个方案最优？
   - 评估维度：时间、成本、风险、效果
   - 选择：方案 X

5. 如何执行？
   - Step 1: ...
   - Step 2: ...
   - Step 3: ...

6. 执行后如何验证？
   - 验证指标：[具体指标]
   - 验证方法：[测试/Review/用户反馈]
```

## 🛡️ 自我兜底机制

### 自主修复循环（3次尝试）

```
测试失败
    ↓
[第1次] 诊断 → 定位 → 调研 → 方案 → 修复 → 验证
    ↓ 失败
[第2次] 换思路 → 修复 → 验证
    ↓ 失败
[第3次] 回滚 → 通知人工
```

**完全自主权限**（类似 Oh My OpenCode）：

- ✅ 可以修改代码
- ✅ 可以切换方案
- ✅ 可以重启进程
- ✅ 可以尝试替代方案
- ✅ 可以调用外部 Agent（Codex、Gemini）

# 测试
cd .worktrees/skill-test

# 清理
git worktree remove .worktrees/skill-test

## 🔒 历史错误记录（避免重蹈覆辙）

### 错误模式 1：过度自动化

- **触发场景**：连续 5 次自动修正被拒绝
- **错误原因**：自动确认阈值过高，忽略了用户真实意图
- **解决方案**：降低自动确认阈值，增加人工确认环节
- **记录时间**：2026-02-28

### 错误模式 2：规则冲突

- **触发场景**：多个 Hook 同时触发，互相干扰
- **错误原因**：未定义 Hook 优先级，导致执行顺序混乱
- **解决方案**：定义 Hook 优先级系统（Block > Warn > Info）
- **记录时间**：2026-02-28
